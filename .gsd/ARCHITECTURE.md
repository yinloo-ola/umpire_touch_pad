## Overview

**Umpire Touchpad** is a full-stack web application that replicates the ITTF Umpire Touchpad — the digital scoring device used by table tennis referees. It supports singles and doubles matches, handling the full match lifecycle from scheduling to completion. It features an **Umpire interface** for live scoring and an **Admin dashboard** for match management and real-time monitoring.

## Components

### Umpire Interface
- **MatchListView.vue**: Displays scheduled matches. Fetches today's matches from `/api/matches`.
- **SetupView.vue**: Pre-match configuration (sides, server, doubles quadrants, warmup timer).
- **TouchpadView.vue**: Main scoring interface. Handles points, lets, game advances, and winner confirmation.
  - **UX Rule**: Player names are hidden when status is "Start of Play".
- **CardModal.vue**: Component for issuing Yellow, Yellow-Red, and Red cards to players or coaches.
- **CardIndicators.vue**: Visual indicator on the touchpad for issued cards.
- **TimeoutModal.vue**: 60-second countdown timer for called timeouts.

### Admin Dashboard
- **AdminLayout.vue**: Wraps admin views with consistent navigation and auth checks.
- **LoginView.vue**: Secure login for administrators using JWT-based authentication.
- **DashboardView.vue**: Live overview of matches. Supports "Show History" to view past matches.
- **MatchDetailView.vue**: Deep dive into a single match, showing full score history and issued cards.
- **MatchFormView.vue**: Interface for creating or editing match details (players, best-of, type).

### matchStore.js (Pinia)
The core engine of the application. Responsible for:
- **Scoring Logic**: Automatic game-over detection and server rotation.
- **ITTF Rules**: Implementation of singles/doubles serve cycles (A→X→B→Y→A) and deuce handling.
- **Volatile State**: Quadrant positions (`p1Top`, etc.), side swaps, and "Start of Play" flags.
- **Card System**: Tracks issued cards per team/coach. Awards penalty points automatically (YR1=1pt, YR2=2pts).
- **Sync Engine**: `syncMatch()` snapshots the entire store to the backend on every change.
- **State Recovery**: `fetchMatchState()` rehydrates the store from `state_json` in the DB, enabling exact resume after refresh.

### Go Backend
- **internal/api**: REST handlers. Uses `RequireAuth` middleware for admin routes.
- **internal/service**: Bridges JSON data to DB models. `MatchService` handles the complex mapping of nested match/game/card data.
- **internal/store**: Generated by `sqlc`. Provides type-safe access to SQLite.
- **db/migrations**: Managed by `goose`. Version-controlled schema including tables for `matches`, `games`, and `cards`.

## Data Flow

1. **Admin/Service** creates matches via POST `/api/match`.
2. **Umpire** selects match → `fetchMatchState` loads current points/cards/volatile state.
3. **Umpire Action** (Point, Card, Timeout) → Store updates locally → `syncMatch` called.
4. **Backend Sync**: POST `/api/match` updates `matches.status`, `matches.state_json`, and syncs `games` and `cards` tables.
5. **Dashboard Refresh**: Admin view polls or reloads `/api/matches?history=true` to reflect live umpire state.

## Conventions

- **State Is Sacred**: Components must never mutate store state directly. Use actions.
- **Snapshot Persistence**: The entire match state is treated as a snapshot for synchronization.
- **Atomic Actions**: Actions like `nextGame` or `issueCard` wrap multiple state changes.
- **Mobile First**: Umpire interface optimized for 10-inch tablets (large touch targets).
