# Plan 2.1: Backend Schema and API for Admin Match Editing

## Objective
Implement a single-endpoint backend that allows an admin to overwrite a match's full state (status, games, and remarks), enforcing score validation unless a remark is provided, and auto-reindexing games.

- .gsd/SPEC.md

- backend/db/migrations/
- backend/db/query.sql
- backend/internal/service/match_svc.go
- backend/internal/api/handlers.go

<task type="auto">
  <name>Match Service Admin Update Logic</name>
  <files>
    backend/internal/service/match_svc.go
  </files>
  <action>
    - Define a new struct `AdminMatchUpdateRequest` in `match_svc.go` (Status string, Remarks string, Games []SyncGameRequest).
    - Implement `AdminUpdateMatch(ctx context.Context, matchID string, req AdminMatchUpdateRequest) error` inside `MatchService`.
    - In the transaction:
      1. Re-index games: loop through `req.Games`, enforcing sequential `GameNumber` (1, 2, ...).
      2. Strict Score Validation: Verify if game scores are valid (e.g., reached 11 with 2 point difference). If the game score isn't valid, it must reject the request UNLESS `req.Remarks` is non-empty (indicating a Force End/Retirement).
      3. Update match status & remarks via `AdminUpdateMatch` query.
      4. Clear all existing games for this match (Need a new query `DeleteGamesForMatch` or use raw sql if not generated).
      5. Insert the re-indexed games via `UpsertGame`.
  </action>
  <verify>grep -q "AdminUpdateMatch" backend/internal/service/match_svc.go</verify>
  <done>AdminUpdateMatch method is fully implemented with validation and re-indexing</done>
</task>
